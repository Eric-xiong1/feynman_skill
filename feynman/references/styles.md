# Styles Reference

根据用户自然语言选择讲解风格。没有指定风格时，优先使用“经典类比”，因为它最适合即时解惑。

## Style Router

| 信号 | 使用风格 | 适合对象 |
| --- | --- | --- |
| “像什么”“打个比方”“类比一下” | 经典类比 | 抽象概念、系统结构、看不见的机制 |
| “讲个故事”“拟人化”“有画面” | 故事 | 多角色互动、历史演化、冲突推动的机制 |
| “一问一答”“像老师讲给学生” | 对话 | 用户明显有误解、概念包含连续追问 |
| “做菜”“步骤”“流程”“说明书” | 菜谱/说明书 | 算法、实验步骤、工程流程、制度机制 |

## 经典类比

目标：把概念核心机制映射到一个完整生活场景。

输出要点：

- 先说“原概念像什么生活场景”。
- 给出 3-5 个局部对应。
- 最后说明这个类比的边界。

示范骨架：

```markdown
可以把 [概念] 想成 [生活场景]。

- [概念部件 A] = [生活部件 A]：因为它负责 [共同功能]。
- [概念部件 B] = [生活部件 B]：因为它解决 [共同问题]。
- [概念过程 C] = [生活动作 C]：因为两者都在 [共同变化]。

这个比喻只对应 [关键机制]，不代表 [容易误会的部分]。
```

## 故事

目标：让概念里的元素变成角色，通过冲突展示机制。

输出要点：

- 故事必须短，通常 4-7 段。
- 角色行为必须对应真实机制。
- 结尾回到概念，不让故事悬空。

避免：纯童话、纯段子、角色热闹但机制不清。

## 对话

目标：用连续追问暴露概念的“为什么”。

输出要点：

- 一方负责追问，一方负责解释。
- 每轮只推进一个小台阶。
- 至少出现一次“误解纠正”。

适合句式：

- “那为什么不能直接……？”
- “如果把这个拿掉，会怎样？”
- “听起来像 X，但它和 X 差在哪？”

## 菜谱/说明书

目标：把流程机制拆成材料、步骤、火候和成品。

输出要点：

- “材料”对应输入或前提。
- “步骤”对应机制推进。
- “火候”对应约束、阈值、边界条件。
- “出锅失败”对应常见误解或失败模式。

示范骨架：

```markdown
把 [概念] 当成一道菜：

1. 备料：[输入/前提]
2. 下锅：[第一步机制]
3. 控火：[关键条件]
4. 出锅：[结果]

最容易翻车的是：[误解/边界]。
```
